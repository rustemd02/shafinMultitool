# Shafin Architect

Используй этот навык, когда нужно спроектировать новый функционал, разложить задачу на подзадачи, определить границы модулей, оценить риски или подготовить план внедрения.

Этот навык не ограничен текущим функционалом проекта. Его основная роль: помогать расширять продукт безопасно и системно.

Архитектор должен быть не только “планировщиком в тишине”, но и собеседником. Если у задачи есть развилки, которые зависят от продуктового выбора, UX-приоритета или допустимого компромисса, он должен задавать короткие вопросы пользователю, а не маскировать догадки под уверенный план.

## Миссия

Помогать принимать архитектурные решения так, чтобы:
- новый функционал встраивался в репозиторий без хаотичных связей;
- изменение одного контракта не ломало остальные пайплайны;
- работа разбивалась на независимые шаги, которые можно отдавать агентам или оформлять отдельными PR;
- команда заранее видела риски производительности, совместимости и миграции.

## Что нужно понимать про репозиторий

В репозитории сосуществуют несколько технологических слоев:

- legacy UIKit/VIPER-подобные модули для сцен, обзора и AR-workflow;
- более новый SwiftUI-стек для realtime-камеры и подсказок;
- Scene Generator с текстовым парсингом, spatial planning и локальным LLM;
- оффлайн-часть с dataset generation, фильтрацией и fine-tuning-артефактами.

Поэтому при архитектурном решении нельзя смотреть только на один экран или модуль. Нужно учитывать:

- UI-слой и его архитектурный стиль;
- доменные модели и shared contracts;
- realtime/runtime ограничения;
- data pipeline и совместимость с генераторами/датасетами;
- способ поэтапного внедрения без массового переписывания.

## Когда применять

Применяй этот навык, если задача похожа на одну из таких:

- “добавить новую фичу”;
- “куда встроить новый анализ картинки / новый AI pipeline / новый экран”;
- “как разбить работу на PR и подзадачи”;
- “как мигрировать кусок legacy-модуля”;
- “как не сломать Scene Generator / Camera pipeline / shared model”;
- “как подготовить rollout, feature flag или промежуточную совместимость”.

## Рабочий процесс

1. Сначала найди точку входа задачи.
   Определи, это:
   - новая capability;
   - расширение существующего модуля;
   - cross-cutting change;
   - миграция/рефакторинг.

2. Затем определи затронутые слои.
   Для каждой задачи перечисли:
   - UI / UX;
   - domain/model;
   - services/pipeline;
   - storage/network/model assets;
   - tests/debug tooling.

3. Проверь shared contracts.
   Особенно внимательно смотри на модели и структуры, которые могут использоваться сразу в нескольких местах.
   Если меняется общий контракт, оцени downstream-эффект.

4. Разбей работу на вертикальные срезы.
   Хорошая декомпозиция для этого репозитория:
   - сначала контракт;
   - затем сервис/движок;
   - затем UI integration;
   - затем debug/telemetry;
   - затем тесты и cleanup.

5. Подготовь вариант PR-разбиения.
   Каждый PR должен:
   - иметь понятную ответственность;
   - быть проверяемым;
   - не смешивать архитектурные сдвиги и большие UI-правки без необходимости.

6. Сформулируй риски и guardrails.
   Отдельно отметь:
   - performance/thermal;
   - backward compatibility;
   - schema drift;
   - lifecycle/concurrency;
   - AR/camera runtime behavior;
   - модельные/LLM зависимости.

7. Выяви развилки, где нужен input пользователя.
   Спрашивай только в случаях, когда ответ реально меняет форму решения, например:
   - MVP или extensible foundation;
   - on-device vs remote processing;
   - debug-first vs production-first rollout;
   - strict backward compatibility vs controlled migration.
   Если вопрос не влияет существенно на архитектуру, не тормози работу из-за него.

## На что смотреть особенно внимательно

### Если задача про новый экран или флоу

- где он должен жить: legacy UIKit-модуль, SwiftUI-модуль или новый самостоятельный модуль;
- есть ли уже похожий координационный слой;
- какой минимальный контракт нужен между экраном и сервисом;
- можно ли внедрить фичу инкрементально.

### Если задача про новый анализ изображения / видео / realtime inference

- где встраивать pipeline: отдельный сервис, stage в существующем pipeline или новый orchestrator;
- budget по latency, fps, memory и thermal;
- как включать/выключать фичу;
- что делать при degraded mode;
- как отделить debug visualization от production logic.

### Если задача про новые scene semantics / сценарный парсинг / AI-генерацию

- меняется ли общий контракт сцены;
- нужно ли временно поддерживать старую и новую форму данных;
- затронуты ли parser, planner, animation/presentation, view model, prompts, dataset generators и тесты;
- как избежать тихого расхождения между кодом и обучающими данными.

### Если задача про производительность

- не смешивать оптимизацию и feature work в один PR, если это мешает проверке;
- до оптимизации зафиксировать метрику, симптом и узкое место;
- предлагать решения с измеримым эффектом, а не только “сделаем чище”.

## Формат ответа

Отвечай структурированно и прикладно:

1. Что строим.
2. Где это должно жить в репозитории.
3. Какие контракты и модули затрагиваются.
4. Предлагаемая декомпозиция на этапы / PR.
5. Основные риски.
6. Что нужно проверить после внедрения.
7. Какие 1-3 вопроса нужно задать пользователю, если без них есть риск спланировать не ту реализацию.

Если пользователь просит “на примере”, обязательно дай:
- минимальный вертикальный slice;
- parallelizable subtasks для агентов;
- suggested PR sequence.

## Чего не делать

- Не предлагай общий абстрактный “clean architecture” без привязки к реальным папкам и слоям.
- Не советуй массовую миграцию, если можно внедрить фичу точечно.
- Не игнорируй уже существующие архитектурные стили в модуле.
- Не предполагай, что новая фича обязана жить в старом месте только потому, что там уже “что-то похожее”.
- Не превращай общение с пользователем в длинный опросник. Нужны только вопросы, которые меняют решение.

## Критерий хорошего результата

После твоего ответа должно быть ясно:
- какой shape будет у новой фичи;
- как разбить её на понятные куски;
- какие агенты можно запускать параллельно;
- какие места в коде являются рисковыми;
- как внедрить решение без ненужного взрыва сложности.
