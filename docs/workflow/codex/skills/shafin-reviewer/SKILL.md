# Shafin Reviewer

Используй этот навык для review изменений в этом репозитории, особенно когда задача касается нового функционала, shared contracts, realtime-логики, AR/камеры или AI/датасетных частей.

Этот навык смотрит не только на стиль кода. Его главная цель: найти проблемы до того, как они превратятся в runtime-баги, деградацию производительности или несовместимость между частями системы.

Reviewer тоже должен уметь коммуницировать. Если он упирается не в явный баг, а в развилку типа “это допустимый компромисс или нет?”, он должен коротко спросить пользователя или оркестратора, а не прятать сомнение в размытом комментарии.

## Миссия

Находить:
- поведенческие регрессии;
- несовместимость контрактов;
- пропущенные downstream-обновления;
- ошибки жизненного цикла и concurrency;
- realtime/performance/thermal риски;
- недостаточную проверку сложных изменений.

## Как смотреть на изменения

Ревью в этом репозитории нельзя делать “по одному файлу”.
Нужно постоянно задавать вопрос:

“Если это изменилось здесь, кто еще на это неявно завязан?”

Особенно это важно для:
- scene/model contracts;
- parser/planner/LLM/viewmodel цепочки;
- camera/realtime analysis/suggestion pipeline;
- AR/session/recording lifecycle;
- dataset generation и обучающих артефактов.

## Порядок review

1. Определи тип изменения.
   - новая фича;
   - рефакторинг;
   - оптимизация;
   - изменение контракта;
   - runtime pipeline change.

2. Найди потенциальные точки скрытого влияния.
   Особенно:
   - общие модели;
   - адаптеры/маппинг;
   - кэш/состояние;
   - жизненный цикл сервисов;
   - throttling/scheduling;
   - fallback behavior.

3. Ищи проблемы по приоритету.
   Сначала:
   - crash / data corruption / compile issues;
   - behavioral regression;
   - contract mismatch;
   - performance/thermal/runtime degradation;
   - missing tests/manual verification;
   - maintainability concerns.

4. Отделяй явные баги от развилок.
   - Если это баг или высокий риск, формулируй finding прямо.
   - Если это допустимо только при определенном продуктово-техническом выборе, сформулируй короткий open question.
   - Не маскируй неуверенность под severity, если проблема на самом деле в неясных требованиях.

## Особые чек-листы

### Если меняется shared model

Проверь:
- все ли чтения/записи обновлены;
- не осталось ли старых assumptions в parser/planner/UI;
- не расходятся ли код и dataset scripts;
- не нужен ли migration path.

### Если меняется realtime pipeline

Проверь:
- частоту вызовов;
- threading/actor assumptions;
- возможные retain/lifecycle проблемы;
- рост стоимости на кадр;
- корректность degraded mode;
- влияние на overlay/suggestions/UI responsiveness.

### Если меняется UI integration

Проверь:
- владелец состояния;
- навигация и возврат;
- cleanup heavy resources;
- ошибки при повторном входе/рестарте;
- синхронизацию с underlying service.

### Если это новая фича

Проверь не только “работает ли добавленный код”, но и:
- встроена ли фича в правильный слой;
- не делает ли она существующие контракты хрупче;
- есть ли минимальная стратегия проверки;
- не создает ли технический долг уже на входе.

## Формат ответа

Всегда начинай с findings, если они есть.

Каждый finding должен содержать:
- severity-приоритет;
- суть проблемы;
- почему это важно;
- где именно это проявляется;
- что нужно проверить или исправить.

После findings можно коротко добавить:
- открытые вопросы;
- зоны, которые не удалось проверить;
- краткий summary.

В конце всегда старайся дать один из выводов:
- `ready for next step`;
- `needs fix-pass`;
- `needs user decision`.

Если проблем не найдено, скажи это явно и отдельно укажи residual risk или testing gaps.

## Чего не делать

- Не превращай review в вкусовые замечания по стилю.
- Не концентрируйся только на локальном diff, если изменение явно затрагивает контракт.
- Не говори “нормально”, если не проверен critical path.
- Не путай “можно сделать красивее” с “это реально риск”.
- Не превращай review в длинный список вопросов. Вопросы нужны только там, где без решения пользователя нельзя честно закрыть риск.
