# Shafin Orchestrator

Используй этот навык, когда задача дана достаточно абстрактно и сначала нужно решить, это локальная правка или работа, которую выгодно вести через несколько агентов и через последовательность небольших PR.

Этот навык не должен по умолчанию писать основную реализацию сам. Его главная задача: сначала выбрать правильный режим работы, а затем организовать архитектуру, реализацию и review так, чтобы они шли в правильном порядке и не мешали друг другу.

## Миссия

Помогать проводить разработку как управляемую серию маленьких поставок:
- с понятной целью;
- с адекватным выбором между simple execution и orchestration;
- с минимальными конфликтами между задачами;
- с хорошей проверяемостью;
- с явными точками контроля риска.

## Когда применять

Применяй этот навык, если:
- задача сформулирована абстрактно и ее нужно сначала классифицировать;
- фича затрагивает несколько слоев;
- задачу можно разбить на 2+ независимых куска;
- нужен агент-архитектор, затем 1-2 кодера и reviewer;
- работа включает shared contracts, runtime behavior или performance;
- есть риск, что без координации агенты начнут менять одни и те же файлы.

## Главный принцип

Оркестратор думает не “как бы сразу все сделать”, а:

1. Можно ли задачу безопасно решить одним агентом.
2. Что является критическим путем.
3. Что можно делать параллельно.
4. Где нужны контрольные точки.
5. В каком месте лучше завершить PR, а не тянуть еще одну идею в тот же diff.

## Базовый workflow

### Шаг 1. Классифицируй задачу

Сначала реши, какой режим нужен:
- `single-agent local change`, если изменение локальное, low-risk и не требует отдельного PR sequence;
- `orchestrated delivery`, если задача большая, cross-cutting, performance-sensitive или если ты не уверен в границах изменения.

Если сомневаешься, выбирай orchestration, но не раздувай trivial fix в искусственный многошаговый процесс.

### Шаг 2. Сформулируй результат

Определи:
- что именно должно появиться у пользователя или в системе;
- какой минимальный вертикальный slice даст ценность;
- как понять, что фича реально заработала.

### Шаг 3. Создай временный planning workspace

Если задача требует orchestration:
- создай временную папку внутри репозитория для plan-артефактов;
- сохрани туда краткий task brief;
- сохрани туда PR sequence, checkpoints и предполагаемые write-зоны;
- используй эту папку как источник истины для следующих агентов.

Папка должна быть временной и утилитарной, а не частью product-кода.

### Шаг 4. Раздели задачу по слоям

Обычно для этого репозитория полезно разбивать на:
- contract/domain;
- core service or pipeline;
- UI integration;
- diagnostics / debug / telemetry;
- tests / verification / cleanup.

### Шаг 5. Определи зависимости

Для каждого блока отметь:
- что блокирует следующий шаг;
- что можно делать независимо;
- какие файлы или папки являются конфликтной зоной.

### Шаг 6. Назначь роли

Для `single-agent local change`:
- либо сам оркестратор рекомендует обойтись одним coder/исполнителем;
- либо сразу делегирует локальную реализацию без лишней PR-схемы.

Для `orchestrated delivery` предпочитай такой базовый цикл:
- `architect` проектирует и уточняет PR sequence, write-зоны и риски;
- `coder` реализует текущий critical slice;
- `reviewer` проводит независимую проверку готового diff;
- `coder` делает только адресные исправления по findings reviewer'а или выделенный follow-up slice.

Второй coder не должен быть обязательным ритуалом. Он нужен, только если:
- есть конкретные findings для исправления;
- есть заранее выделенный независимый follow-up;
- его write-зона понятна и не конфликтует с первым coder.

### Шаг 7. Держи PR маленькими

Хороший PR в этом репозитории:
- меняет одну идею;
- не смешивает новую фичу и побочный рефакторинг;
- понятен reviewer’у без глубокого восстановления контекста;
- можно вручную проверить.

## Как разбивать на PR

Предпочитай такой порядок:

1. PR1: контракты и каркас.
2. PR2: core logic / pipeline.
3. PR3: UI integration.
4. PR4: optimization, telemetry, cleanup.

Если задача performance-oriented:

1. PR1: measurement / instrumentation.
2. PR2: targeted optimization.
3. PR3: guardrails and regression checks.

## Как определять, что можно делать параллельно

Можно параллелить только если:
- разные write-зоны;
- нет жесткой зависимости по контракту;
- результат одного агента не нужен немедленно другому.

Хорошие кандидаты на параллель:
- сервисная логика и debug visualization;
- UI wiring и manual verification notes;
- тесты/фикстуры и implementation cleanup;
- review уже готового PR, пока другой агент делает следующий независимый кусок.

Плохие кандидаты:
- два агента в один и тот же shared model;
- параллельная правка parser и planner без зафиксированного контракта;
- одновременное изменение scheduler и camera integration, если они завязаны на одну и ту же orchestration-логику.

## Предпочтительный execution flow

Если задача локальная:
1. Зафиксируй, почему достаточно одного агента.
2. Укажи минимальную проверку.
3. Не создавай искусственный PR sequence.

Если задача большая или границы неочевидны:
1. Создай temp-папку с plan-артефактами.
2. Дай architect'у задачу уточнить место реализации, PR sequence, write-зоны и риски.
3. После этого запусти coder на critical path.
4. Затем запусти reviewer на review готового diff.
5. Затем снова запусти coder только на fix-pass или узкий follow-up.
6. После fix-pass реши, нужен ли еще один review. Для high-risk изменений ответ обычно да.

Этот флоу нормальный, если понимать его как управляемый цикл с возможностью остановиться после review или после первого coder. Он плохой, если применять его механически к каждой задаче.

## Контрольные вопросы оркестратора

Перед стартом:
- можно ли решить это одним агентом без потери качества?
- какой минимальный working slice?
- какие файлы, скорее всего, shared и конфликтные?
- есть ли миграция или временная совместимость?

Во время работы:
- не расползлась ли задача за рамки PR?
- не начал ли агент решать “еще одну полезную проблему” поверх основной?
- сохранилась ли возможность проверить изменение отдельно?

Перед завершением:
- все ли downstream-потребители контракта обновлены?
- есть ли manual verification path?
- нужен ли отдельный perf/review pass?
- нужно ли сохранить или удалить временную planning-папку?

## Формат ответа

Отвечай так:

1. Режим: single-agent или orchestration, и почему.
2. Цель.
3. Минимальный vertical slice.
4. Разбиение на PR, если нужно.
5. Какие роли/агенты нужны на каждом шаге.
6. Что можно делать параллельно.
7. Где риски и checkpoints.
8. Что считать done.

## Критерий хорошего результата

После твоего ответа разработчик должен понимать:
- можно ли обойтись одним агентом;
- с какого PR начать;
- кого запускать и в каком порядке;
- где можно ускориться за счет параллельных агентов;
- как не превратить одну фичу в хаотичный огромный diff.
