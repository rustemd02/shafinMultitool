# 30. Semantic Camera Analysis Source of Truth

Status: canonical source-of-truth for the next Camera Analysis stage.
Date: 2026-05-21.

Этот документ фиксирует целевую систему после уточнения требований. Если старые документы расходятся с ним по продуктовой цели, UX, роли моделей или демонстрационным сценариям, этот документ имеет приоритет. Детальные контракты и PR-дизайны остаются в документах `03-29`.

## 1. Core Thesis

Цель системы: приложение должно в реальном времени и в режиме паузы объяснимо подсказывать пользователю, как сделать кадр красивее.

Ключевой пользовательский эффект:
- человек наводит камеру;
- приложение показывает короткую live-подсказку;
- пользователь нажимает паузу;
- приложение объясняет, что в кадре хорошо, что мешает, почему система так решила, насколько она уверена и что конкретно сделать.

Ключевая исследовательская идея:
- live остается быстрым, локальным и стабильным;
- pause дает более глубокий explainable-разбор;
- тяжелое visual reasoning выносится в ограниченный VLM/teacher-контур или в будущую compact on-device semantic model;
- финальный пользовательский совет всегда выбирается из закрытого каталога действий, а не генерируется свободным текстом.

## 2. Defense-Level Positioning

На защите комиссия должна увидеть не просто camera-coach UI, а комплексную computer science систему:

- real-time mobile computer vision pipeline;
- fusion deterministic features + optional neural semantic evidence;
- bounded recommendation planner;
- confidence estimation;
- explainability trace;
- eval dataset с expected и forbidden tips;
- demo/debug mode, показывающий, из каких сигналов возник совет.

Выбранный научно-инженерный акцент: `hybrid cascade`.

Это сильнее, чем просто «подключить модель», потому что показывает архитектурное разделение:
- fast local live path;
- pause analysis path;
- optional VLM teacher path;
- future on-device distilled student;
- deterministic bounded planner;
- measurable quality gates.

## 3. Product Scope

### In Scope

- Live показывает один главный совет.
- Live совет короткий, мягкий и не назойливый.
- Live может хвалить хороший кадр и объяснять, почему он хорош.
- Pause показывает расширенный разбор: что хорошо, что плохо, что сделать.
- Confidence обязателен.
- Debug/defense mode включается отдельной кнопкой.
- Система должна заметно отличаться от простого camera coach за счет семантики: объект + причина + действие + уверенность + объяснение.

### Non-Goals For This Stage

- Полноценный генеративный ассистент, который пишет любые советы свободным текстом.
- Remote VLM в live.
- Глубокая режиссерская постановка сцены.
- Полноценный multi-person cinematography solver.
- Гарантированное понимание всех объектов и всех жанров фотографии.
- Production privacy/security model.

## 4. Live UX Contract

Live режим должен показывать только один главный совет.

Требуемые live-типы:
- сдвинуть камеру левее;
- сдвинуть камеру правее;
- сдвинуть камеру выше;
- сдвинуть камеру ниже;
- изменить свет;
- подойти ближе или отойти;
- выровнять горизонт;
- оставить кадр как есть, если он уже читается хорошо;
- object-aware tip при высокой уверенности, например: «Цветок мешает лицу» или «Лампа отвлекает от героя».

Generic background advice не является обязательным первым live-MVP. Но object-aware interference tip обязателен как «вау-эффект», если система уверенно видит объект и может связать его с проблемой кадра.

Если уверенность низкая, live должен формулировать осторожно:
- «Возможно, стоит чуть сдвинуть камеру правее».
- «Похоже, лицо теряется в тени».
- «Кажется, объект на переднем плане отвлекает».

Confidence в live показывается визуально:
- цвет;
- иконка;
- тон бейджа.

Проценты в live не нужны.

## 5. Pause UX Contract

Pause должен быть главным explainability-режимом.

Структура pause-карточки:
1. Вердикт.
2. Что в кадре хорошо.
3. Что мешает кадру.
4. Что сделать.
5. Confidence у каждого совета.
6. Краткое evidence/debug объяснение по отдельной кнопке или раскрытию.

Обязательные категории pause-разбора:
- композиция;
- свет;
- фон;
- объектная съемка;
- «почему кадр хороший».

Работа с несколькими людьми не является отдельной обязательной категорией pause-разбора, но система должна fail-safe обрабатывать два и более лица:
- считать их группой;
- не выбирать случайное одно лицо без уверенности;
- при низкой уверенности не выдавать категоричные советы.

Confidence в pause показывается процентами у каждого значимого элемента:
- вердикт;
- проблема;
- сильная сторона;
- действие.

## 6. Required Demo Scenarios

Приоритетные сценарии для защиты:

- портрет одного человека;
- предметная съемка;
- кадр из фильма или экрана;
- пейзаж или улица;
- темный кадр;
- пересвет или окно за спиной;
- хороший кадр, который приложение должно не испортить.

Второй приоритет:
- два человека в кадре;
- захламленный фон;
- object-aware distraction, например цветок, лампа, чашка или телефон мешают главному субъекту.

Кадр из фильма является обязательным сценарием. Система должна уметь признать, что кадр уже может быть композиционно сильным, и не пытаться механически исправлять каждый кадр.

## 7. Semantic Tip Definition

Семантическая подсказка считается качественной, если в ней есть четыре слоя:

- объект: к чему относится совет;
- причина: что именно ухудшает или улучшает кадр;
- действие: что пользователь может сделать физически;
- уверенность: насколько система уверена.

Пример плохой подсказки:
- «Камеру правее».

Пример целевой подсказки:
- «Попробуй сдвинуть камеру чуть правее: лицо сейчас близко к краю, и кадру не хватает воздуха слева».

Пример object-aware подсказки:
- «Похоже, цветок отвлекает от лица. Попробуй сместить кадр чуть правее или убрать его из переднего плана».

## 8. Supported Advice Catalog

Минимальный закрытый каталог действий:

- `moveFrameLeft`;
- `moveFrameRight`;
- `moveFrameUp`;
- `moveFrameDown`;
- `increaseSubjectSize`;
- `decreaseSubjectSize`;
- `improveFrontLight`;
- `addBackgroundLight`;
- `reduceBacklight`;
- `levelHorizon`;
- `reduceObjectDistraction`;
- `preserveCurrentFrame`;
- `explainGoodComposition`.

Желательные cinematic extensions:

- `addRimLight`;
- `increaseDepthSeparation`;
- `preserveLookSpace`;
- `rebalanceGroupSubject`;
- `avoidOvercorrectingCinematicFrame`.

Extensions можно проектировать заранее, но не нужно выдавать пользователю без надежного evidence.

## 9. Model Strategy

### Current Direction

Основа должна быть offline/on-device.

Remote VLM должен быть подготовлен как pause-only extension point, но не должен быть обязательным для работы приложения.

### Why A Model Is Needed

Новая модель или VLM-teacher нужен не для генерации красивого текста, а для visual-semantic evidence:

- лицо теряется на фоне;
- объект конкурирует с главным субъектом;
- кадр уже кинематографично скомпонован;
- свет отделяет или не отделяет героя от фона;
- фоновой свет улучшит глубину;
- два субъекта читаются как группа;
- объект в кадре создает смысловой шум.

Эти признаки трудно надежно вывести только из DETR, Vision face detection, horizon и exposure heuristics.

### Preferred Architecture

1. Deterministic live path.
2. Pause deterministic critique.
3. Optional pause VLM evidence provider.
4. Bounded fusion.
5. Closed action catalog.
6. Future compact on-device semantic evidence model, обученная на teacher/eval данных.

### VLM Boundary

VLM не должен:
- писать финальный пользовательский текст;
- создавать произвольные действия;
- работать в live;
- ломать deterministic fallback;
- быть единственным source of truth.

VLM должен:
- вернуть структурированное evidence;
- указать confidence;
- связать evidence с регионом, объектом или субъектом;
- выбрать action ids из закрытого каталога;
- позволить deterministic planner принять или отклонить его вклад.

## 10. Explainability Contract

Каждый совет должен восстанавливаться до цепочки:

`signal -> observation -> interpretation -> recommendation -> UI`.

Пример:
- signal: лицо найдено слева, subject edge pressure высокий;
- observation: лицо близко к левому краю;
- interpretation: кадру не хватает пространства для чтения героя;
- recommendation: сдвинуть камеру чуть левее или правее в зависимости от ориентации и look-space;
- UI: короткая подсказка + стрелка + confidence.

Для защиты нужно уметь показать:
- какие детекторы участвовали;
- какие признаки были рассчитаны;
- почему совет выбран, а другие отброшены;
- почему confidence именно такой;
- почему система промолчала при низкой уверенности.

## 11. Evaluation Requirements

Eval должен проверять не только «есть совет», но и качество поведения:

- expected live tip;
- expected pause tips;
- forbidden tips;
- minimum confidence;
- object grounding;
- whether silence is correct;
- whether good frame is preserved;
- whether cinematic frame is not overcorrected.

Минимальный набор:
- 25-30 кадров для eval;
- 5-7 кадров для защиты;
- минимум 3 хороших кадра, где система должна не навязывать исправления;
- минимум 3 кадра из фильма или экрана;
- минимум 3 световых проблемы;
- минимум 3 object-aware distraction случая.

## 12. What Is Needed From The User

Нужно собрать небольшой, но качественный датасет.

От пользователя нужны:
- изображения или ссылки на изображения;
- короткое объяснение, почему кадр красивый или некрасивый;
- желаемый пользовательский совет;
- что система не должна советовать;
- отметка, является ли кадр demo-priority;
- если кадр из фильма, указать, что важно сохранить: свет, композицию, позу, фон, настроение.

Формат, в котором удобно присылать:

```text
Кадр 1:
Тип: портрет / предмет / фильм / пейзаж / световая проблема
Почему хороший или плохой:
Что хочется, чтобы приложение посоветовало:
Что нельзя советовать:
Приоритет для защиты: да/нет
```

Я беру на себя:
- превращение этих описаний в labels;
- формирование JSONL fixtures;
- expected/forbidden tip mapping;
- confidence thresholds;
- eval cases;
- agent prompts;
- дальнейшую декомпозицию на PR.

## 13. Definition Of Done

Функция считается готовой для продукта, если:

- live показывает один стабильный, полезный совет;
- live умеет осторожно формулировать low-confidence советы;
- live умеет не советовать лишнего;
- pause показывает расширенный разбор;
- pause содержит объяснение, уверенность и действия;
- object-aware tip работает хотя бы на ограниченном наборе объектов;
- хороший кадр не портится механическими советами;
- fallback на deterministic path всегда работает.

Функция считается готовой для защиты, если:

- есть 5-7 сильных демонстрационных сценариев;
- есть eval-набор 25-30 кадров;
- есть debug/defense mode;
- можно показать pipeline от pixels/signals до recommendation;
- можно объяснить, почему architecture mobile-first;
- можно объяснить, зачем нужен VLM teacher;
- можно показать, что VLM bounded и не заменяет весь алгоритм черным ящиком;
- можно показать limitations честно.

## 14. Open Decisions

Эти решения пока не надо закрывать до сбора первых кадров:

- точный список object-aware объектов для MVP;
- нужен ли real remote VLM в прототипе или хватит mock/template + контрактов;
- нужно ли тренировать compact student model до защиты или достаточно подготовить distillation pipeline;
- какие метрики будут основными: precision подсказок, stability, forbidden-tip rate, explainability completeness или latency.

## 15. Strong Recommendation

Следующий правильный шаг не код, а dataset/eval alignment.

Если начать сразу подключать VLM, система может стать эффектной, но недоказуемой. Для диссертации сильнее сначала зафиксировать:
- какие кадры считаются сложными;
- какие советы считаются правильными;
- какие советы запрещены;
- как измеряется уверенность;
- где deterministic core достаточен;
- где нужен VLM/semantic model.

После этого можно безопасно писать новый PR pipeline.
