# Chapter 3. Генерация раскадровки сцены

В данной главе рассматривается модуль генерации раскадровки сцены, предназначенный для преобразования сценарного описания в структурированную модель. Это представление необходимо для последующего отображения превизуализации в среде дополненной реальности.

В качестве входных данных пользователь загружает текст сценария или отдельный фрагмент сцены. Входные данные подаются в произвольной форме и могут включать в себя различные детали мизансцены:

- имена персонажей;
- реплики;
- обозначение локации;
- указания на действия актёров;
- служебные заметки, надписи на экране.

Загруженный сценарий преобразуется в структурированное представление. Оно должно содержать элементы, значимые для построения раскадровки и описанные в тексте сценария естественным языком. На основе полученного представления формируется превизуализация сцены в среде дополненной реальности.

## 3.1. Формирование структурированного представления сцены

Перед началом работы необходимо зафиксировать контракт сцены. Помимо сущностей, явно описанных в сценарии (персонажи, реплики и т. д.), контракт должен содержать технические элементы, необходимые для корректного построения раскадровки.

Основные поля контракта сцены представлены в таблице 3.1.

| Элемент | Назначение |
|---|---|
| Актёры (`actors`) | Действующие лица сцены. |
| Объекты (`objects`) | Элементы сцены, включающие объекты из сценария, а также объекты, вручную размеченные пользователем в среде дополненной реальности. |
| Действия (`actions`) | Действия актёров внутри сцены. В рамках раскадровки они отображаются в виде физического изменения состояния или положения виртуальных объектов. Например, фрагмент «Иван идёт к Лёше» содержит действие «идёт», которое интерпретируется как перемещение виртуальной модели одного персонажа к модели другого. В программном решении поддерживается ограниченный набор действий: перемещение, остановка, взаимодействие с предметами, на которые можно сесть, взгляд на объект, взятие и размещение предмета, передача предмета другому персонажу. |
| Описанные действия (`described_actions`) | Действия или состояния актёров, которые не входят в набор заданных физических действий, но необходимы для сохранения смысловой информации сценария. Например, действия «Иван маринует мясо» или «Лёша отжимается» отображаются на виртуальной сцене в виде текстовой подсказки. |
| Сценический такт (`beat`) | Смысловая единица времени внутри сцены, объединяющая одно или несколько одновременно происходящих действий или изменений состояния. Такты необходимы для поддержки корректной хронологии развития событий внутри сцены. |
| Пространственные отношения (`spatial_relations`) | Относительное расположение одного элемента сцены относительно другого: «у окна», «под столом», «рядом с Лёшей». При создании раскадровки такие отношения служат основой для первичного размещения объектов на виртуальной сцене. |
| Реплики (`dialogue`) | Разговор персонажей внутри сцены, отображаемый на раскадровке как текстовый элемент. |
| Метаданные сцены (`scene_metadata`) | Общие сведения о сцене: время действия, тип сцены (интерьерная или экстерьерная), название или локация сцены. |
| Текстовые элементы (`text_elements`) | Части сценария, которые не могут быть представлены в виде физических объектов на виртуальной сцене. Например: «надпись на экране: „Спустя 3 часа“», «звонок в дверь». |

Особую роль в контракте играет сценический такт. Например, фрагмент «Иван открывает дверь на кухню, Лёша сидит в темноте. Иван идёт к Лёше и говорит: „Что ты здесь делаешь?“» может быть представлен как два последовательных такта. Первый такт содержит действия: Иван открывает дверь, Лёша сидит. Второй такт содержит действия: Иван идёт к Лёше, Иван говорит. Такое разбиение позволяет сохранить не только сами действия, но и порядок их выполнения. Для раскадровки это принципиально, поскольку одинаковый набор действий при разной последовательности может описывать разные сценические ситуации.

Для оценки качества структурированного представления использовались метрики, отражающие не только формальную корректность JSON, но и пригодность результата для построения раскадровки:

- `json_valid_rate` — доля ответов модели, которые могут быть разобраны как корректный JSON;
- `schema_valid_rate` — доля ответов, соответствующих ожидаемому контракту сцены;
- `object_count_match_rate` — доля примеров, в которых модель корректно выделила количество объектов сцены;
- `beat_count_match_rate` — доля примеров, в которых модель корректно восстановила количество сценических тактов;
- `action_count_match_rate` — доля примеров, в которых модель корректно восстановила количество действий.

Эти метрики были выбраны потому, что каждая из них связана с отдельным условием построения раскадровки: JSON должен быть технически читаемым, структура должна соответствовать контракту, такты должны сохранять хронологию, а действия — определять изменение состояния или положения виртуальных элементов.

### 3.1.1. Первый подход: прямое формирование описания сцены

Первый подход к формированию структурированного описания сцены был основан на прямом преобразовании сценария на естественном языке в структуру, которая использовалась бы для построения раскадровки. Другими словами, задача сводилась к генерации JSON-объекта с заданной структурой по входному тексту (рис. ПЛЕЙСХОЛДЕР). Подобная постановка относится к распространённой практике использования языковых моделей для генерации структурированного вывода [ПЛЕЙСХОЛДЕР].

Рисунок ПЛЕЙСХОЛДЕР — Схема прямого формирования структурированного описания сцены

Для уменьшения задержки между вводом и выводом решения, а также для сохранения автономности приложения, было принято решение использовать компактную языковую модель, запускаемую локально на мобильном устройстве. Такой подход в том числе важен для сценарных материалов, содержащих закрытую информацию, а также делает поведение системы независимым от качества сетевого соединения и внешней инфраструктуры [ПЛЕЙСХОЛДЕР].

В качестве базовой модели была выбрана модель `Qwen3-1.7B` [ПЛЕЙСХОЛДЕР]. Выбор обусловлен тем, что эта модель является instruction-моделью, то есть ориентирована не только на свободное продолжение текстового ввода, но и на выполнение конкретных системных инструкций. Для уменьшения требований к вычислительным ресурсам был использован квантизированный вариант модели. Эта и последующие модели были сконвертированы в формат `GGUF` и интегрированы в Swift-приложение при помощи библиотеки `llama.cpp` [ПЛЕЙСХОЛДЕР].

В первом подходе работа модели задавалась системным промптом. В нём фиксировалась роль модели как парсера сценического описания, задавалась инструкция по формированию JSON-объекта из текста и определялись ограничения на поведение модели: сохранять хронологию, использовать допустимые типы действий, не добавлять события, отсутствующие в сценарии, и следовать заданной структуре JSON. Полный системный промпт первого подхода представлен в приложении ПЛЕЙСХОЛДЕР.

Сформированный моделью JSON далее проходил несколько уровней проверки. На первом уровне проверялась синтаксическая корректность JSON. На втором уровне результат сопоставлялся с ожидаемой структурой сцены: проверялось наличие основных секций, корректность идентификаторов, допустимость типов действий и наличие обязательных полей. На третьем уровне оценивалась пригодность результата для построения раскадровки: действия должны были ссылаться на существующих актёров и объекты, сценические такты должны были сохранять порядок событий, а неподдерживаемые действия должны были переноситься в текстовое описание. Если проверка выявляла синтаксическую ошибку, нарушение структуры или некорректные связи между элементами сцены, результат не использовался напрямую и должен был быть заменён или дополнен fallback-механизмом.

Схематическое изображение формирования описания сцены базовой моделью представлено на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР — Формирование описания сцены базовой языковой моделью

При этом первичная проверка показала, что синтаксическая корректность JSON сама по себе не гарантирует пригодность результата для построения раскадровки. Базовая модель могла формировать формально валидный JSON, однако часто нарушала ожидаемую структуру сцены: представляла массивы актёров и объектов в неверном формате, объединяла несколько сценических тактов в один, пропускала действия или описывала их свободным текстом вместо использования поддерживаемых типов. Поэтому дальнейшая работа была направлена не только на получение валидного JSON, но и на повышение соответствия результата контракту сцены.

Следующим шагом стало дообучение базовой модели под конкретный формат сцены, при этом архитектура представления сцены оставалась прежней. Для дообучения модели использовалась схема прямого отображения:

```text
source_text -> SceneScript JSON
```

где `source_text` — исходный текст сценария, `SceneScript JSON` — корректное структурированное представление сценария.

Для дообучения модели был сформирован обучающий набор, содержащий `1850` элементов, а его основой стали сценарные примеры, покрывающие набор типовых ситуаций из киносцен. Для последующей общей оценки подходов был зафиксирован отдельный тестовый набор из `262` сценарных фрагментов, не входивший в обучающий корпус. Раннее сравнение базовой и дообученной моделей, приведённое ниже, выполнялось на фиксированном наборе из `198` фрагментов.

Для оптимизации процесса дообучения компактной модели были рассмотрены методы `LoRA` и `QLoRA` [ПЛЕЙСХОЛДЕР]. Применение этих методов позволяет не переобучать все параметры модели, а добавлять небольшой набор адаптируемых весов, что снижает требования к вычислительным ресурсам и делает экспериментальный цикл быстрее.

Результаты тестирования дообученной модели показаны в таблице 3.2.

Таблица 3.2 — Результаты сравнения базовой и дообученной модели на первом подходе

| Метрика | Базовая модель | Дообученная модель |
|---|---:|---:|
| Соответствие контракту структуры | `15/198` (`7,6 %`) | `114/198` (`57,6 %`) |

Дообучение заметно снизило количество структурных ошибок. Число результатов, соответствующих контракту сцены, выросло с `15` до `114` из `198` проверенных фрагментов, а число нарушений схемы снизилось со `183` до `84`. При этом полное совпадение с эталонным представлением было достигнуто только в `24` случаях из `198`. Это показывает, что дообучение улучшило воспроизведение формата, но не решило задачу точного восстановления содержания сцены.

Несмотря на повышение структурной устойчивости вывода, прямой подход сохранил существенное ограничение: целевой JSON мог быть формально валиден, но семантически не соответствовать тексту, что вызвало необходимость в пересмотре архитектуры решения.

### 3.1.2. Второй подход: каноническое описание сцены

Первый подход показал, что компактная локальная модель недостаточно точно справляется с комплексной задачей анализа сценарного текста, выделения сущностей, восстановления хронологии и последующего формирования JSON. Поэтому следующей задачей стал перенос части работы из этапа генерации моделью в этап предварительной подготовки обучающих данных.

Основная идея второго подхода заключается в предварительном формировании канонического описания сцены. До формирования обучающей пары создаётся сцена, в которой явно заданы все элементы: персонажи, объекты, действия, сценические такты и связи между сущностями. Затем из этого канонического описания формируются два связанных представления: текстовое описание сцены и целевое структурированное представление сцены. Схема формирования обучающей пары представлена на рисунке ПЛЕЙСХОЛДЕР.

```text
Библиотека сценических паттернов
        ↓
Детерминированный генератор сцены
        ↓
Каноническое описание сцены
        ↓                         ↓
Текстовое описание сцены      Целевой SceneScript JSON
        ↓                         ↓
             Обучающая пара SFT
```

Рисунок ПЛЕЙСХОЛДЕР — Формирование обучающей пары через каноническое описание сцены

Создание набора канонических сцен выполняется при помощи разработанного генератора. Для этого была сформирована библиотека сценических паттернов, в которой заданы типовые классы сцен: движение персонажей друг к другу, передача предмета, диалог, а также комплексные сцены с комбинацией нескольких действий. На основе выбранного класса генератор формирует экземпляр сцены с заданным числом персонажей, набором объектов и последовательностью сценических тактов. Для воспроизводимости генерация выполняется с предварительно заданным seed-значением: один и тот же класс сцены при одинаковых параметрах приводит к одному и тому же каноническому описанию.

Далее для каждого экземпляра канонической сцены формируется набор текстовых описаний, а также целевой JSON, используемый как эталонный ответ при обучении модели. Текстовое описание создаётся в трёх стилях: расширенном, нейтральном и сокращённом. Расширенный вариант содержит дополнительную незначимую информацию, нейтральный описывает сцену без дополнительных деталей, а сокращённый оставляет только элементы, необходимые для создания корректной раскадровки. Это необходимо, поскольку одна и та же сцена может быть описана пользователем разными словами и с разной степенью подробности.

Целевой JSON формируется детерминированным преобразователем канонической сцены в структуру контракта описания сцены. Такой способ позволяет сохранить детали исходного описания и не требовать от языковой модели самостоятельного восстановления всех связей между объектами, актёрами и действиями на этапе подготовки эталонного ответа.

Таким образом, каноническая сцена становится источником истины для обеих сторон обучающей пары: текстового входа и целевого структурированного представления. В отличие от первого подхода, где обучающий пример фактически задавал только соответствие «текст — итоговый JSON», второй подход сначала фиксирует семантику сцены, а уже затем порождает из неё разные представления. Это снижает вероятность того, что модель будет обучаться на противоречивых или неполных целевых ответах.

Следующим шагом стало дообучение модели методом предпочтений `Odds Ratio Preference Optimization` (`ORPO`) [ПЛЕЙСХОЛДЕР]. В отличие от традиционного дообучения методом `SFT`, где модель обучается воспроизводить корректный ответ на паре «ввод — вывод», дообучение методом предпочтений использует пару ответов разного качества: предпочтительный и ошибочный. В ошибочном примере фиксировались типовые ошибки модели: нарушение порядка тактов, неверная привязка действия к цели, пропуск действия или некорректное использование объекта. Схема дообучения методом предпочтений представлена на рисунке ПЛЕЙСХОЛДЕР.

```text
Текстовое описание сцены
        ↓
Ответ модели с ошибкой        Канонический SceneScript JSON
        ↓                              ↓
Отклоняемый ответ              Предпочтительный ответ
        ↓                              ↓
             Пара предпочтений
                    ↓
             ORPO-дообучение
```

Полученные результаты показывают, что каноническое описание сцены существенно повышает структурную устойчивость вывода: модель чаще формирует валидный JSON и результат, соответствующий контракту сцены. Дообучение методом предпочтений дало небольшой прирост по метрикам, связанным с семантической корректностью: корректность определения цели действия выросла с `6,32 %` до `8,89 %`, корректность хронологии — с `4,58 %` до `6,49 %`, а полнота восстановления действий — с `6,03 %` до `8,42 %`. Однако абсолютные значения метрик, связанных с действиями, целями и хронологией, оставались недостаточными для надёжного построения раскадровки. Это показало, что даже при более качественном обучающем корпусе компактная модель перегружена задачей прямого формирования итогового `SceneScript`.

### 3.1.3. Третий подход: промежуточный план сцены

Предыдущий подход позволил повысить структурную устойчивость за счёт изменения процесса подготовки обучающих данных, однако сохранил ограничения компактной модели, которой всё ещё было необходимо решать ряд комплексных задач, включая понимание текста и формирование итоговой структуры.

Третий подход направлен на модификацию целевого представления, которое генерирует модель, а также на декомпозицию архитектуры решения. Вместо прямого формирования итогового JSON модель должна генерировать промежуточный план сцены `ScenePlanIR`, который в дальнейшем при помощи детерминированного компилятора преобразуется в итоговое представление сцены `SceneScript` по зафиксированному контракту.

Ключевое отличие плана сцены от итогового контракта заключается в разделении смыслового и технического уровней описания сцены. `SceneScript` является итоговым представлением, содержащим технические данные, необходимые для корректного построения раскадровки. `ScenePlanIR`, в свою очередь, является внутренним упрощённым представлением, содержащим смысловую структуру сцены: актёров, объекты, такты, действия, пространственные отношения и привязки ссылок.

В качестве второго слоя подхода выступает детерминированный компилятор. Его задача заключается в проверке и исправлении набора типовых ошибок языковой модели: нормализации ссылок на персонажей и объекты, проверке связей между элементами сцены и формировании итоговых тактов. Например, если модель указала действие, для которого требуется цель, но не задала ссылку на объект или персонажа, компилятор заменяет такое действие на нейтральное состояние актёра и фиксирует диагностическую информацию `v8.targetless_action_downgraded` для последующего анализа.

Обучающие данные для третьего подхода продолжают формироваться на основе канонических сцен. Однако теперь из канонического описания формируется не только итоговый `SceneScript`, но и промежуточный `ScenePlanIR`. Целевым ответом для модели становится промежуточный план, а итоговое представление используется как результат его детерминированной компиляции.

Схема подхода с промежуточным планом сцены отображена на рисунке ПЛЕЙСХОЛДЕР.

```text
      ↓
Текстовое описание сцены

      ↓
ScenePlanIR

```

## 3.2. Формирование итогового представления сцены

Несмотря на высокие показатели четвёртого подхода, выход модели остаётся вероятностным результатом генерации, поэтому между моделью и модулем визуализации необходим дополнительный детерминированный слой. Его задача заключается в проверке структурированного описания, исправлении типовых ошибок и преобразовании промежуточного представления в итоговый контракт сцены.

Первым этапом после генерации является проверка таблицы событий. Для каждой строки проверяется соответствие справочнику слотов: исполнитель действия должен ссылаться на существующий `actorSlot`, цель действия — на допустимый `targetSlot`, а такт — на один из заранее сформированных `beatSlot`. Если в таблице обнаруживается ошибка, система пытается исправить её только в тех случаях, когда исправление может быть выполнено однозначно. Например, если строка содержит действие с неверным идентификатором такта, оно может быть перенесено в ближайший допустимый такт. Если модель указала неизвестный слот объекта или актёра, такая ссылка блокируется, поскольку её нельзя безопасно сопоставить с элементом сцены.

Для сохранения контролируемости все исправления сопровождаются диагностическими кодами. Примеры таких кодов представлены в таблице 3.8.

Таблица 3.8 — Примеры диагностических кодов при проверке таблицы событий

| Диагностический код | Значение |
|---|---|
| `v9.beat_slot_mismatch` | Событие ссылается на некорректный такт. |
| `v9.action_type_repaired` | Тип действия был приведён к допустимому значению. |
| `v9.target_slot_repaired` | Ссылка на цель действия была исправлена или удалена. |
| `v9.holding_slot_repaired` | Ссылка на удерживаемый объект была исправлена или удалена. |
| `v9.targetless_event_repaired` | Действие без обязательной цели было преобразовано в безопасное представление. |
| `v9.unknown_slot_blocked` | Неизвестная ссылка была отклонена. |

При преобразовании таблицы событий выполняется разрешение ссылок. Поле `actorSlot` сопоставляется с актёром, `targetSlot` — с актёром или объектом, а `holdingObjectSlot` — с объектом, который находится у персонажа. Это позволяет ограничить модель от добавления произвольного персонажа или объекта, отсутствующего в исходном сценарии.

Далее промежуточный план передаётся в компилятор `ScenePlanCompiler`. Компилятор выполняет нормализацию идентификаторов: порядковые ссылки на актёров (`first`, `second`, `third`) преобразуются в стабильные идентификаторы вида `actor_n`, а объекты получают идентификаторы вида `object_k`. Объекты, размеченные пользователем в среде дополненной реальности, сохраняют связь с исходной разметкой.

После компиляции результат проходит проверку качества при помощи алгоритма на основе правил. Алгоритм определяет, можно ли использовать локально полученный `SceneScript` для построения раскадровки. В зависимости от результата проверки система выбирает один из маршрутов обработки:

- `accept_local` — результат принят и используется для построения раскадровки;
- `needs_clarification` — описание содержит неоднозначность, требующую уточнения пользователем;
- `fallback_rule_only` — результат модели непригоден, поэтому используется rule-based fallback.

Fallback-политика необходима для того, чтобы не использовать сомнительный результат модели напрямую. Если структурированное описание формально разобрано, но содержит критическую ошибку, например неразрешённую ссылку на объект или нарушение структуры тактов, система выбирает более безопасный маршрут обработки.

## 3.3. Сборка сцены из фрагментов сценария

При работе с пользовательским сценарием приложение не может рассматривать входной текст как один цельный запрос к модели. В реальном сценарном фрагменте могут одновременно присутствовать заголовки сцен, реплики, ремарки, действия, надписи на экране и несколько последовательных фаз одной сцены. Кроме того, локальная компактная модель ограничена по размеру контекста и вычислительным ресурсам мобильного устройства [ПЛЕЙСХОЛДЕР]. Поэтому передача всего сценария в модель целиком увеличивает задержку обработки и повышает риск потери связей между персонажами, объектами и действиями.

Для решения этой задачи в приложении используется система деления исходного сценария на фрагменты. Основная идея заключается в том, что исходный сценарий делится на сцены и чанки, после чего каждый чанк обрабатывается отдельно, но с учётом состояния уже разобранной части сцены. Итоговое представление формируется не прямой генерацией одной большой структуры, а последовательной сшивкой фрагментов. Схема системы представлена на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР — Общая схема сборки сцены из фрагментов сценария

Первым этапом является нормализация сценарного текста. Компонент `ScriptNormalizer` приводит переносы строк к единому виду и разбивает текст на нормализованные строки `NormalizedScriptUnit`. Каждая строка получает тип: `scene_heading`, `speaker_cue`, `dialogue`, `stage_note`, `screen_text`, `action_line`, `prose_line` или `blank`. Такое разделение необходимо, потому что одна и та же строка сценария может иметь разный смысл для последующей обработки: реплика должна быть связана с говорящим персонажем, ремарка может быть сохранена как текстовая подсказка, а строка действия должна попасть в модель как физическое или описанное действие.

В нормализаторе также обрабатываются частые сценарные конструкции. Строка вида «Иван: Я всё понял» разделяется на `speaker_cue` и `dialogue`. Ремарки внутри реплики, записанные через символ `*`, выделяются отдельно как `stage_note`. Длинные реплики дополнительно разбиваются на несколько фрагментов, чтобы один чанк не становился слишком крупным.

После нормализации выполняется выделение сцен. Компонент `SceneBoundaryDetector` ищет заголовки сценария и извлекает из них метаданные. Поддерживаются форматы `INT`, `EXT`, `INT/EXT`, а также русскоязычные варианты `ИНТ`, `ЭКСТ`, `ИНТ/ЭКСТ`, `НАТ`. Из заголовка определяется локация, тип сцены и время суток. Если заголовок отсутствует, весь переданный текст рассматривается как одна неявная сцена. В результате формируется набор `ScriptSceneCandidate`, где каждая сцена содержит собственный диапазон строк, исходный текст и метаданные.

Далее каждая сцена делится на чанки при помощи `ChunkSegmenter`. Чанк является минимальной единицей обработки моделью и содержит:

- `sceneID` — идентификатор сцены;
- `chunkID` — идентификатор фрагмента;
- `chunkIndex` — порядковый номер фрагмента внутри сцены;
- `sourceText` — исходный текст чанка;
- `sourceRange` — диапазон символов в исходном документе;
- `metadata` — метаданные сцены.

Сегментация зависит от типа строк. Например, если чанк начинается с `speaker_cue`, он обрабатывается как диалоговый фрагмент и включает последующие строки реплики до пустой строки, нового говорящего или нового заголовка сцены. Такое деление позволяет не смешивать в одном запросе к модели слишком много независимых событий. Фрагмент с репликой персонажа и последующее действие после слова «затем» будут обработаны как разные чанки, но останутся внутри одной сцены и затем будут сшиты в единую хронологию. Пример преобразования нормализованных строк сценария в чанки представлен на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР — Пример разбиения сцены на чанки

Для сохранения непрерывности между чанками используется `SceneChunkState`. Он формируется из текущего состояния сцены и передаётся в локальный планировщик вместе с текстом очередного чанка. В состояние входят уже известные персонажи и объекты, алиасы персонажей и объектов, последний разрешённый говорящий, позы персонажей и удерживаемые предметы. Благодаря этому следующий фрагмент обрабатывается не как изолированный текст, а как продолжение уже начатой сцены. Например, если в первом чанке персонаж берёт папку, а во втором сказано «он передаёт её Лёше», состояние сцены позволяет передать модели информацию о том, какие персонажи и объекты уже существуют и какие предметы находятся у персонажей. Это снижает вероятность того, что модель создаст нового актёра или новый объект вместо использования уже существующих элементов. Использование состояния между двумя последовательными чанками показано на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР — Передача состояния сцены между чанками

Результат обработки отдельного чанка сначала рассматривается как локальный черновик. Он может содержать ссылки, корректные только внутри данного фрагмента, например `first`, `second`, `object_1`. Поэтому перед сшивкой используется `ChunkCanonicalizer`. Его задача — преобразовать локальные ссылки в стабильные ссылки сцены.

Канонизация выполняет несколько операций. Для актёров проверяются имена, алиасы и порядковые ссылки `first`, `second`, `third`. Если актёр уже был известен по предыдущим чанкам, новый локальный идентификатор связывается с существующим стабильным идентификатором. Для объектов используется похожий механизм: объект сопоставляется с уже известным объектом по имени, типу, алиасу или связи с размеченным AR-объектом. Объекты, которые не упомянуты в тексте и не используются действиями, могут быть отброшены как галлюцинации модели.

Особое значение имеет обработка неразрешённых ссылок. Если действие ссылается на объект или персонажа, которого нельзя безопасно сопоставить с существующим элементом сцены, такая ссылка не подменяется произвольным значением. Она либо переносится как отложенная ссылка, либо удаляется из действия, если не влияет на смысл. Если действие требует цели, но цель не может быть разрешена, оно может быть преобразовано в описанное действие, чтобы сохранить смысл исходного текста без создания некорректной физической траектории. Логика преобразования локальных ссылок в стабильные ссылки сцены и обработки неразрешённых ссылок показана на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР — Канонизация и обработка неразрешённых ссылок внутри чанка

После канонизации каждый чанк представляется как набор изменений к сцене. Объект `SceneChunk` содержит новые актёры и объекты, новые такты, новые пространственные отношения, изменения состояния и диагностические коды. Это позволяет обрабатывать сшивку не как объединение готовых JSON-объектов, а как последовательное применение изменений к состоянию сцены.

Сшивку выполняет `SceneStitcher`. Он последовательно применяет чанки к `SceneStitchState`. При добавлении чанка обновляется реестр актёров и объектов, добавляются новые такты, объединяются пространственные отношения и обновляется состояние сцены. Если несколько чанков содержат одно и то же действие, оно не добавляется повторно: для действий используется fingerprint, включающий актёра, тип действия, цель, реплику и исходный текст. Это защищает итоговую сцену от дублирования событий при повторном упоминании одной и той же ситуации.

Во время сшивки также обновляются runtime-состояния: текущие позы актёров, удерживаемые предметы, последняя известная позиция актёра и краткое описание предыдущего чанка. Эти данные используются не только для итоговой сцены, но и для обработки следующих фрагментов. Таким образом, пайплайн работает инкрементально: каждый новый чанк уточняет состояние сцены и влияет на разбор последующих чанков. Последовательное применение чанков к состоянию сцены показано на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР — Сшивка чанков в итоговый план сцены

После обработки всех чанков сцены `SceneStitcher` формирует финальный `ScenePlanIR`. На этом этапе отбрасываются действия с несуществующими актёрами, очищаются некорректные ссылки на объекты, а пространственные отношения фильтруются по существующим элементам сцены. Если действие с обязательной целью осталось с неразрешённой ссылкой, оно переводится в безопасное текстовое описание. Это позволяет сохранить смысл сценария, но не строить ошибочную AR-анимацию.

Затем для всего документа формируется `SceneBundlePlan`. Он содержит набор сцен, их чанки, диагностическую информацию и визуальные элементы, извлечённые из сценария. После этого `SceneBundleCompiler` компилирует каждую сцену в `SceneScript`, а итоговым результатом становится `SceneBundleScript`. Для интерфейса приложения используется активная сцена `activeSceneScript`, однако внутри сохраняется структура всего документа, включая список сцен, чанков и диагностик.

Отдельно поддерживается режим добавления текста. Если пользователь дописывает сценарий в конец, система не обязана заново пересобирать весь документ. В режиме `append` переиспользуются ранее сформированные сцены, чанки и состояния. Если новый текст является продолжением последней неявной сцены, пересобирается только эта сцена. Если в добавленном фрагменте появляется новый заголовок сцены, он добавляется как новая `ScriptSceneCandidate`. Такой подход снижает объём повторной обработки и делает сценарное редактирование более пригодным для интерактивного использования.

Таким образом, система чанков решает две задачи одновременно. С одной стороны, она ограничивает объём текста, передаваемого локальной модели за один вызов. С другой стороны, она сохраняет непрерывность сцены за счёт реестра сущностей, состояния чанков и детерминированной сшивки. Итоговый `SceneBundleScript` получается не как независимая генерация по каждому фрагменту, а как результат последовательной сборки сценарного документа в единую структуру, пригодную для последующего построения AR-раскадровки.

## 3.4. Построение AR-раскадровки

Построение раскадровки в дополненной реальности выполняется после завершения генерации и компиляции структурированного представления. В интерфейсе приложения используется активная сцена `activeSceneScript`, полученная из `SceneBundleScript`. Далее эта сцена передаётся в слой пространственного планирования. На основе текущего положения камеры, обнаруженных плоскостей, размеченных пользователем объектов и данных из `SceneScript` формируется объект `PlannedScene`. Он содержит уже не только смысловую структуру, но и данные, пригодные для непосредственного отображения: позиции актёров и объектов, начальные повороты, траектории движения, длительности сегментов и текстовые аннотации для воспроизведения. Общая схема построения AR-раскадровки представлена на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР – Общая схема построения AR-раскадровки

Система построения раскадровки в дополненной реальности включает в себя следующие основные этапы:

- получение итогового `SceneScript`;
- сопоставление объектов сценария с размеченными объектами реального пространства;
- расчёт пространственного плана сцены в `SpatialPlannerService`;
- создание 3D-сущностей RealityKit;
- запуск воспроизведения действий по тактам сцены.

В качестве основы механизмов работы с дополненной реальностью используются фреймворки ARKit [] и RealityKit []. Сцена дополненной реальности создаётся в компоненте `ARSceneContainer`, который представляет собой SwiftUI-обёртку над `ARView`. Для управления AR-сессией используется `ARSessionDelegate`, а конфигурация сессии задаётся явно. Автоматическая конфигурация `ARView` отключена при помощи параметра `automaticallyConfigureSession = false`, так как приложению необходимо самостоятельно управлять режимами AR-сессии: обычным режимом отображения сцены, режимом разметки объектов и паузой во время генерации.

Отдельно обрабатывается состояние генерации. Во время работы локальной языковой модели AR-сессия временно приостанавливается. Это связано с тем, что одновременная работа ARKit, RealityKit и локальной модели увеличивает нагрузку на CPU, GPU и память мобильного устройства. Схема управления AR-сессией представлена на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР – Управление AR-сессией при построении раскадровки

Для построения раскадровки приложению необходимо получать не полный поток AR-кадров, а ограниченный набор данных о текущем состоянии пространства. Эти данные извлекаются из `ARFrame` в методе `session(_:didUpdate:)`. С целью оптимизации обработка кадров ограничивается частотой 15 Гц. При этом это не влияет на итоговую частоту кадров записанного видеоролика.

Из каждого обработанного кадра извлекаются следующие данные:

- `cameraTransform` – матрица положения и ориентации камеры в мировом пространстве;
- `cameraIntrinsics` – внутренняя матрица камеры, используемая при восстановлении 3D-точки из экранной координаты;
- `imageResolution` – разрешение изображения камеры;
- `planeAnchors` – обнаруженные горизонтальные плоскости;
- `depthMap` – карта глубины, используемая только в режиме ручной разметки объектов.

Для связи сценарного описания с реальным пространством в приложении используется ручная разметка объектов. Этот механизм нужен в тех случаях, когда в тексте сценария упоминается физический объект, уже находящийся перед пользователем: стол, дверь, шкаф, стул и т. д. Если такой объект не связать с координатой в AR-пространстве, система сможет создать только виртуальный объект-заглушку, но не сможет корректно использовать реальный предмет как цель действия.

Разметка выполняется в отдельном режиме. После включения режима пользователь нажимает на объект в кадре, а приложение пытается определить 3D-позицию точки нажатия. Сначала используется карта глубины, если она доступна. Для этого в AR-сессию временно добавляется поддержка `sceneDepth` или `smoothedSceneDepth`, после чего из последнего AR-кадра сохраняется компактный снимок `DepthFrameSnapshot`: карта глубины, матрица камеры, внутренняя матрица камеры и разрешение изображения.

При обработке нажатия экранная координата переводится в координату карты глубины. Затем из `depthMap` считывается значение глубины. Значение считается допустимым только в ограниченном диапазоне: оно должно быть больше нуля и меньше 10 метров. После этого выполняется восстановление 3D-точки в системе координат камеры:

```text
cameraX = (imageX - cx) * depth / fx
cameraY = (imageY - cy) * depth / fy
cameraZ = -depth
```

где `fx`, `fy`, `cx`, `cy` берутся из матрицы `cameraIntrinsics`. Полученная точка в системе координат камеры умножается на `cameraTransform`, после чего получается позиция объекта в мировых координатах AR-сцены [ПЛЕЙСХОЛДЕР].

После размещения объектов планировщик рассчитывает начальные позиции актёров. В отличие от объектов, актёры почти всегда являются виртуальными сущностями, поскольку их задача – показать мизансцену и перемещение персонажей внутри кадра. Начальное размещение актёров зависит от действий, содержащихся в сцене. В планировщике используется несколько сценариев размещения:

- если в действиях присутствует направление `toward_each_other`, то первые два актёра размещаются на противоположных сторонах сцены относительно её центра. Такое размещение используется для сцен, где персонажи идут навстречу друг другу;
- если в сцене есть действия типа `approach` с заданной целью, начальная позиция актёра рассчитывается относительно этой цели. Если несколько актёров идут к одному и тому же объекту, они распределяются в ряд перпендикулярно направлению к цели. Это предотвращает наложение моделей друг на друга и сохраняет читаемость мизансцены;
- если специальных условий нет, актёры размещаются в ряд относительно центра сцены. При этом начальный поворот каждого актёра рассчитывается так, чтобы модель была направлена к центру сцены. Для этого из позиции актёра строится вектор к центру `SceneSpace`, а угол поворота вокруг вертикальной оси вычисляется через `atan2`.

Схема начального размещения актёров представлена на рисунке ПЛЕЙСХОЛДЕР.

Рисунок ПЛЕЙСХОЛДЕР – Начальное размещение актёров в AR-сцене

Построение траекторий выполняется по тактам `SceneBeat`. Все действия внутри одного такта рассматриваются как события, происходящие в один временной интервал. Для каждого актёра планировщик выбирает действия текущего такта и последовательно преобразует их в изменения позиции или текстовые аннотации. После обработки всех актёров вычисляется длительность такта: она равна максимальной длительности действий среди актёров, но также учитывает поле `minDuration`. При этом слишком большие значения `minDuration`, полученные от модели, ограничиваются верхней границей 4 секунды, чтобы воспроизведение не выглядело как зависшее. Если один актёр в текущем такте выполняет более короткое действие, чем другой, к его траектории добавляется дополнительная точка ожидания в текущей позиции. Благодаря этому все актёры синхронизируются по длительности такта, а следующий такт начинается одновременно для всей сцены. Основные варианты преобразования действий в AR-поведение представлены в таблице ПЛЕЙСХОЛДЕР.

Таблица ПЛЕЙСХОЛДЕР – Преобразование действий в AR-поведение

Для перемещения к цели планировщик сначала определяет позицию цели. Целью может быть как объект, так и другой актёр. Если `target` указывает на объект, позиция берётся из массива `placedObjects`. Если `target` указывает на актёра, используется позиция этого актёра на начало текущего такта. Такой механизм позволяет обрабатывать не только фразы вида «подходит к столу», но и фразы вида «подходит к Лёше».

Действие `approach` реализуется как движение к целевой позиции с остановкой перед ней. Планировщик строит вектор от текущей позиции актёра к цели, нормализует его и выбирает финальную точку на расстоянии около 0,5 м от цели. Это нужно, чтобы актёр не попадал внутрь объекта или модели другого персонажа.

Действие `pass_by` строится как обход цели. Сначала рассчитывается точка рядом с объектом со смещением 0,8 м в перпендикулярном направлении, затем точка за объектом со смещением вдоль направления движения. В результате актёр не просто приходит к объекту, а визуально проходит мимо него.

Для действий с направлением `left`, `right`, `forward`, `backward`, `away_from_each_other` используется смещение относительно локальных осей `SceneSpace`. Например, движение вправо рассчитывается не по глобальной оси `X`, а по вектору `right`, вычисленному относительно текущего положения камеры. Благодаря этому направление движения остаётся согласованным с перспективой пользователя.

Длительность каждого сегмента движения вычисляется на основе расстояния и скорости действия:

```text
duration = distance / speed
```

Скорость зависит от типа действия и модификатора. Для ходьбы и подхода используется базовая скорость 0,8 м/с, для бега – 2,5 м/с.

Важно, что не каждый тип действия из контракта сцены превращается в отдельное физическое движение. Некоторые действия, например сложные бытовые или не поддерживаемые в текущем AR-планировщике, сохраняются как текстовая аннотация или ранее переводятся в `described_action`. Это позволяет не строить неверную анимацию, если действие нельзя безопасно представить имеющимися средствами.

После размещения сущностей пользователь может запустить воспроизведение сцены. Каждый актёр анимируется последовательно по точкам своей траектории. Для каждого сегмента берутся начальная точка, конечная точка и длительность. Перед началом движения модель поворачивается в сторону направления движения. Угол рассчитывается по вектору между начальной и конечной точкой:

```text
angle = atan2(delta.x, delta.z)
```

где `delta` – вектор от начальной точки сегмента к конечной точке.

Для записи видеопотока программное решение получает кадры из `ARSession` через метод `session(_:didUpdate:)`. В качестве видеоданных используется `frame.capturedImage`, то есть исходный пиксельный буфер камеры. Этот буфер передаётся в `AVAssetWriterInputPixelBufferAdaptor` и записывается в `.mov` файл. Для видео используется кодек HEVC, а параметры разрешения и частоты кадров берутся из настроек приложения.

Звук записывается отдельно через `AVCaptureAudioDataOutput`. Аудиосэмплы синхронизируются с начальным временем записи и добавляются в тот же `AVAssetWriter`. Для аудио используется AAC с частотой дискретизации 48 кГц и одним каналом. После остановки записи файл сохраняется в медиатеку устройства через `PHPhotoLibrary`.

Ключевое ограничение этого механизма заключается в том, что `frame.capturedImage` содержит изображение камеры до наложения RealityKit-сущностей. Следовательно, записанный видеопоток не содержит виртуальных актёров, объектов и подписей AR-раскадровки. Такой подход используется для получения «чистой» записи с камеры без AR-компонентов. Виртуальная раскадровка при этом остаётся инструментом превизуализации и постановки сцены на экране устройства, но не запекается в итоговое видео.

Таким образом, построение AR-раскадровки выполняется как отдельный детерминированный этап после генерации структурированного представления сцены. Языковая модель отвечает за извлечение смысловой структуры сценария, а AR-модуль отвечает за пространственное размещение, построение траекторий, создание 3D-сущностей и воспроизведение сцены во времени. Такое разделение позволяет не требовать от модели генерации координат или параметров анимации, а рассчитывать их на стороне приложения с учётом текущего положения камеры, обнаруженных плоскостей и объектов, размеченных пользователем в реальном пространстве.
